Per ridefinire gli attori sonar_adapter e led_adapter seguendo rigorosamente i principi del libro ISS26 (in particolare l'approccio Port/Adapter e la semantica Qak), dobbiamo trasformarli in componenti che rispettano la distinzione tra livello applicativo (core) e livello infrastrutturale.

Secondo il libro, il QActor deve essere un'entità che modella il comportamento come Macchina di Moore, agendo solo sui messaggi di dominio e delegando l'infrastruttura al Context o a supporti esterni.
1. Sonar Adapter (Input Adapter)

Questo attore deve tradurre l'evento infrastrutturale (MQTT) in un evento di dominio (sonardata). Il libro suggerisce di utilizzare un "ponte" che nasconda i dettagli del broker.
Snippet di codice

QActor sonar_adapter context ctxprototype {
    State s0 initial {
        println("sonar_adapter | AVVIATO") color blue
        // Il libro suggerisce di delegare al sistema l'infrastruttura
        // tramite la dichiarazione del broker nel Context o primitive MQTT
        subscribe "sensore/sonar" 
    }
    Goto work
    
    State work { }
    Transition t0 
        // L'evento infrastrutturale arriva dall'infrastruttura MQTT
        whenMsg mqtt_data -> handle_mqtt_payload

    State handle_mqtt_payload {
        onMsg(mqtt_data : msg(TOPIC, PAYLOAD)) {
            [# val Payload = payloadArg(1) #]
            // TRASFORMAZIONE: da messaggio di trasporto a evento di dominio
            // Il cargoservice "vedrà" solo questo evento tipizzato
            emit sonardata : distance($Payload)
        }
    }
    Goto work
}

2. Led Adapter (Output Adapter)

Questo attore funge da "ponte verso l'esterno". Riceve un dispatch applicativo (led_ctrl) e lo traduce nel comando infrastrutturale.
Snippet di codice

QActor led_adapter context ctxprototype {
    State s0 initial {
        println("led_adapter | AVVIATO") color blue
    }
    Goto work
    
    State work { }
    Transition t0 
        whenMsg led_ctrl -> translate_cmd
        
    State translate_cmd {
        onMsg(led_ctrl : ledCmd(CMD)) {
            [# val Command = payloadArg(0) #]
            // TRASFORMAZIONE: da linguaggio di dominio a infrastruttura
            // Il libro spinge ad usare le primitive di pubblicazione
            publish "comando/led" -m $Command
            println("led_adapter | Pubblicato su MQTT: $Command") color blue
        }
    }
    Goto work
}

Note di Allineamento al libro ISS26

Per essere pienamente conforme alla filosofia del Prof. Natali, tieni a mente questi tre punti:

    Anti-Corruption Layer (ACL): Il tuo cargoservice non deve mai contenere stringhe come "MQTT" o riferimenti a socket. Riceve distance(D) e invia ledCmd(CMD). Se domani passi da MQTT a CoAP, cambierai solo il codice interno di questi due attori, lasciando intatto il Core Business.

    Modello a stati (FSM): Gli adattatori che ho scritto sono automi minimi. Poiché la loro unica responsabilità è tradurre (mappare), non devono gestire logica complessa. Se l'adapter dovesse diventare complesso, significherebbe che stai inserendo logica di business nel layer infrastrutturale, violando la Clean Architecture.

    Configurazione del Context: Ricorda che nel file .qak devi dichiarare il broker nel contesto, come prescritto dal libro:
    Snippet di codice

    Context ctxprototype ip [host="localhost" port=8050] {
        mqttBroker "localhost" : 1883 eventTopic "sensore/sonar"
    }
